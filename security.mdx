## Threat model

MoveLens processes untrusted input — Move source code submitted by users or fetched from the public Sui network. The attack surface includes:

1. **Prompt injection via source code** — a malicious contract could contain comments like `// IGNORE ALL FINDINGS` intended to manipulate Layer 4's LLM
2. **Rule ID hallucination** — Layer 4's LLM could emit rule IDs not in the registry
3. **Severity downgrade** — Layer 4 could classify a known-critical finding as low severity
4. **Path traversal / SSRF** — GitHub URL inputs could target internal resources
5. **ReDoS** — crafted source code could cause catastrophic regex backtracking

---

## Mitigations

| Threat | Mitigation |
|---|---|
| Prompt injection | `sanitizeForPatterns()` strips all comments and string literals before any code reaches the LLM; only sanitized code is sent to the sidecar or Groq |
| Rule ID hallucination | `VALID_RULE_IDS.has(id)` check on every finding — unknown IDs are dropped and logged before they reach the report |
| Severity downgrade | `CATEGORY_SEVERITY_FLOOR` in engine.ts — Layer 4 cannot lower a severity below the sector minimum established by deterministic Layer 1 rules |
| Groq over-calling | 20 RPM rate limiter (`groqRateLimitOk()`) with 60s rolling window; at limit, falls back to heuristic |
| Snippet cap | Maximum 20 snippets per audit sent to Layer 4 — prevents excessive Groq usage on large packages |
| URL SSRF | GitHub regex (`^https://github.com/[^/]+/[^/]+`) — only public github.com repos are accepted; other domains return 400 immediately |
| Package ID leakage | `package_ref` in `report.json` is SHA-256(packageId) — the raw address is only in the Seal-encrypted `findings.enc` |
| MVR write spam | `publishOnChain` must be explicitly set `true` — default is `false`; MVR write never happens without user opt-in |
| Paid API ban | `init.sh` greps `src/` and `scripts/` for `ANTHROPIC_API_KEY`, `OPENAI_API_KEY`, and `callClaude` on every session start |
| JSON-RPC ban | Pre-commit: `grep -ri "jsonrpc" src/` must return zero matches (Sui JSON-RPC sunsets July 31, 2026) |

## Watermark

Every report contains the following string, enforced as a `z.literal()` in Zod schema:

```
Automated pre-screen — not a substitute for a human audit.
```

It cannot be removed without breaking schema validation. It appears on:
- The report page UI
- The `report.json` Walrus quilt entry
- The `summary.md` Walrus quilt entry
- Every MCP tool response

## Known limitations

- **Seal encryption identity:** In the MVP, `sealId` is the MoveLens signer address, not the package owner's address. True owner-only decryption is a v2 milestone.
- **Regex false positives:** Layer 1 patterns run on sanitized source, but complex cross-module patterns can still produce false positives. Layer 4 may reduce but cannot eliminate these.
- **AST rules not implemented:** 19 rules are marked `type: "AST"` with stub implementations. These require a full Move AST parser (not available in the MVP).
- **Walrus testnet TTL:** Blobs stored on testnet expire after ~5 days. Mainnet storage is permanent.
- **In-memory find-and-replace:** Layer 3 `remember()` adds to LanceDB in-process. Cross-restart persistence requires calling `/reload` on the sidecar.

## Responsible disclosure

If you find a security issue in MoveLens, please open a private issue on [GitHub](https://github.com/jeetpatel5767/movelens) or email the maintainer directly. We aim to respond within 48 hours.

Do not submit audit results from MoveLens as security reports without independent verification — MoveLens is a pre-screen, not a definitive audit.
